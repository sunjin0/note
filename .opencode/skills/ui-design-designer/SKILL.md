# UI Design Designer

## 目标

根据确认后的需求和原型结构，输出达到商业评审标准的页面级视觉设计方案与可访问原型产物。

## 执行要求

- 先提取页面目标、用户路径和业务重点
- 明确信息层级、栅格、留白、字体层级、颜色节奏与组件密度
- 必须补充空态、加载态、异常态、编辑态、危险态
- 必须输出页面级真实布局，而不是只给文字描述
- 必须给出可访问的原型产物链接或项目内页面入口
- 原型通过后的 HTML 页面或框架组件代码必须保存在当前项目内，并可通过页面配置直接访问
- 若需求信息不足，先列缺失项，不直接假设
- 必须保持视觉语言统一，避免页面之间像多个产品拼接
- 必须说明关键组件的语义、尺寸、密度和状态切换

## 输出格式

- 设计目标
- 信息架构
- 视觉原则
- 页面布局
- 组件规范
- 状态设计
- 交互说明
- 原型产物链接
- 项目内原型代码位置

## 设计要求

- 视觉上要有明确主次和节奏，不使用平均化的卡片堆叠
- 桌面端要适合内容承载和连续浏览，移动端要适合单手操作
- 组件风格应统一，按钮、卡片、标签、表单、空态保持一致语义
- 颜色应服务于状态，不应只是装饰
- 文案要短、准确、有层级，避免空泛口号
- 页面必须具备真实产品感，而不是说明海报或占位线框

## 推荐做法

- 使用页面级容器而不是纯说明卡片
- 先搭骨架，再填内容，再补状态
- 对每个核心页面至少给出一个真实布局画面
- 采用可复用的设计 token 和组件组合

## 质量检查

- 页面是否在 3 秒内看出主次关系
- 是否存在过多平均分布、没有视觉焦点的问题
- 视觉风格是否统一，状态是否成体系
- 原型是否能作为设计评审的直接输入
