## 1. 背景

心情日记是一个面向个人使用的 Web 应用，用于记录每日心情、日记内容、影响因素和照片，并通过日历、仪表盘和统计视图帮助用户了解自己的情绪变化规律。

当前仓库已提供基于 Next.js、React、TypeScript 和 Tailwind CSS 的前端实现，数据主要通过浏览器 `localStorage` 持久化保存。代码中还可见加密、密码保护、认证、同步等扩展能力的基础模块，但 README 仅明确当前版本以本地数据管理为核心，因此本需求文档以“本地优先的心情日记应用”为主线，兼顾现有代码中已经体现的扩展能力边界。

## 2. 目标

1. 让用户能够快速记录每天的心情、日记内容、影响因素和照片。
2. 通过仪表盘、日历和列表视图帮助用户回顾情绪变化与记录习惯。
3. 提供模板、搜索、筛选、提醒和数据管理能力，降低记录成本。
4. 支持多语言、隐私保护和本地数据持久化，满足个人情绪记录场景。

## 3. 范围

### 3.1 本期范围

1. 每日心情记录与编辑。
2. 心情等级管理，包含 5 种心情状态：很棒、不错、一般、难过、生气。
3. 日历视图查看历史记录，包含按日期定位和心情颜色标识。
4. 仪表盘展示连续记录天数、记录总数、最常见心情和趋势信息。
5. 日记列表浏览、搜索、筛选、分组、展开查看、编辑和删除。
6. 影响因素标签管理，支持预设因素与自定义因素。
7. Emoji 选择器，支持分类浏览、搜索和最近使用。
8. 日记模板选择与自定义模板编辑。
9. 智能提醒写日记。
10. 照片上传、预览和删除。
11. 数据导出和本地清空。
12. 简体中文与英文切换。
13. 应用锁、密码设置、安全问题找回与会话控制。
14. 本地数据加密相关能力的基础支持。

### 3.2 代码中已体现但需单独确认纳入节奏的扩展能力

1. 用户认证相关能力。
2. 多设备同步相关能力。
3. 冲突处理与同步状态管理。

## 4. 非范围

1. 统一账号体系下的云端在线协作写作。
2. 多用户共享日记或社交化功能。
3. 服务端强依赖的实时同步发布能力，除非后续单独立项。
4. 复杂的审批流、团队工作台或企业级组织管理能力。
5. 面向桌面客户端的独立实现。
6. 依赖第三方云存储的照片同步与跨端媒体管理。

## 5. 用户场景

1. 用户每天打开应用，快速选择当天心情，补充日记和影响因素后保存。
2. 用户使用模板快速开始写作，减少空白输入成本。
3. 用户上传照片记录当天的情绪相关事件。
4. 用户通过日历查看过去一个月或最近 90 天的心情变化。
5. 用户在仪表盘查看连续记录天数和最近趋势。
6. 用户在列表中按关键词、日期、心情和因素检索历史记录。
7. 用户调整语言、密码、因素和数据导出等个人设置。
8. 用户在连续几天未记录时收到智能提醒。

## 6. 业务规则

1. 一条记录按日期唯一保存，新增同日期记录时应覆盖或更新该日期对应内容。
2. 心情值仅允许在预设的 5 个等级中选择。
3. 日记内容支持富文本编辑，保存时应保留可渲染的 HTML 内容。
4. 影响因素支持多选，既可从预设项中选择，也可添加自定义项。
5. 照片支持多图上传，保存后可在记录中预览与删除。
6. 搜索应覆盖内容、心情、因素与日期等维度。
7. 统计数据应基于本地保存的日记记录实时计算。
8. 用户设置、因素配置、语言和安全信息应保存在本地并保持独立。
9. 数据导出应包含用户的核心记录与配置数据，导出结果应可备份恢复。
10. 密码保护启用后，未通过验证前不应进入应用主体功能。

## 7. 用户场景细化

### 7.1 创建记录

用户从首页、日历或提醒入口打开编辑器，选择心情、填写日记、选择因素、上传照片并保存。

### 7.2 回顾历史

用户在日历中查看带记录日期，点击某天进入编辑；或在列表中按条件筛选后查看详情。

### 7.3 模板写作

用户从模板库选择工作、学习、旅行、健康或生活模板，也可以创建自己的模板并复用变量。

### 7.4 数据管理

用户导出本地数据做备份，或在确认后清空全部记录。

### 7.5 安全访问

用户开启密码保护后，每次进入应用需通过验证；忘记密码时通过安全问题找回。

## 8. 页面与交互说明

1. 首页默认进入仪表盘，提供快速记录入口和统计摘要。
2. 日历页以月视图展示记录，支持月份切换和日期点选。
3. 日记页支持搜索、筛选、分组和分页浏览。
4. 设置页提供语言、因素、数据导出、清空数据和安全相关配置。
5. 记录编辑器以弹窗形式打开，支持表单编辑、照片操作和富文本输入。
6. 侧边栏提供视图切换和移动端导航。

## 9. 异常与边界场景

1. 当当天没有记录时，仪表盘和日历应展示空态提示。
2. 当搜索无结果时，应明确提示无匹配记录。
3. 当照片数量过多或文件过大时，应给出明确限制提示。
4. 当密码错误次数过多时，应进入锁定状态并提示剩余时间。
5. 当本地存储不可用或容量不足时，应提示用户导出或清理数据。
6. 当模板变量无法解析时，应保留可见占位或给出解析失败提示。
7. 当同步相关能力未启用时，不应影响本地记录主流程。

## 10. 验收标准

- [ ] 用户可以创建、编辑、查看和删除心情记录。
- [ ] 记录支持心情、日记、因素和照片字段。
- [ ] 仪表盘、日历和列表三种视图均可正常工作。
- [ ] 可按内容、心情、因素和日期完成检索与筛选。
- [ ] 可选择模板并创建自定义模板。
- [ ] 可添加、编辑、删除和排序自定义因素。
- [ ] 可切换简体中文与英文，且界面文案即时更新。
- [ ] 可启用密码保护并通过安全问题找回。
- [ ] 可导出和清空本地数据，并有二次确认机制。
- [ ] 核心记录数据在刷新后仍可从本地恢复。

## 11. 风险与待确认项

### 11.1 风险

1. 以 `localStorage` 为主的数据方案存在容量上限，照片 Base64 存储会进一步放大体积。
2. 富文本 HTML 内容需要关注 XSS 和内容污染风险。
3. 本地加密和密码保护属于客户端安全能力，防护强度有限。
4. 搜索与统计如果全部在前端实时计算，数据量增大后可能影响性能。
5. 同步与认证能力在代码中已有设计痕迹，但当前需求未明确是否进入本期范围。

### 11.2 已确认处理方式

1. `认证/同步` 不纳入本期正式范围，作为后续独立需求或版本规划处理。
2. `照片存储` 采用 `IndexedDB + Blob` 方案，不再以 `localStorage + Base64` 作为主存储方式，并增加单图和总量限制。
3. `富文本内容` 保存前进行 HTML 白名单清洗，防止脚本注入和恶意标签污染。
4. `密码保护` 定位为本地访问保护能力，保留错误次数锁定和安全问题找回，不承诺强安全防护。
5. `数据导出` 需包含加密状态和恢复说明，便于备份与恢复。
6. `统计口径` 统一定义为：积极心情比例 = `(great + good) / total`，连续记录天数按“有任意记录的连续自然日”计算。
7. `模板与因素国际化` 需要与语言切换联动，保持中英文文案一致。

### 11.3 仍需后续确认的实施细节

1. 照片单张大小上限和单日最大张数。
2. `IndexedDB` 容量接近阈值时的提示文案和清理策略。
3. HTML 白名单的具体标签范围和属性范围。
4. 密码锁定的默认时长和重试次数上限。
